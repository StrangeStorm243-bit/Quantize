## Design decisions resolved here (do not re-litigate in-task)

- **D-a — The value fetch is Inspector-local, not lifted.** A new module-private `ValueBlock`
  component inside `Inspector.tsx` fetches via the shared `useFetch` hook, keyed on the full
  request address. Rationale: unlike the trace tree (shared by the Trace panel + Inspector, hence
  lifted), the value response has exactly one consumer, and its fetch key includes the
  Inspector-local output-port selection. The M14 plan explicitly allowed this
  ("`App.tsx` only if the fetch is lifted") — it is not.
- **D-b — `AtSessionState` gains `runId: string`.** The run layer owns the shape and already
  gates it (`isCursorOnAxis`); the Inspector needs the run id to address the request. Task 2.
- **D-c — Output-port inventory source (a UI default, not numeric logic):** primitive node →
  the catalog `nodeType.outputs` (the same list the Ports section already renders; default =
  first listed); ComponentRef instance → the pinned definition's `exposed_outputs` (default =
  first listed). Unknown node type / definition cache miss → request with `output_port`
  OMITTED and render whatever the server returns (the single-output default, or the served 422
  `ambiguous_output_port` message — honest, no client guessing).
- **D-d — No fetch when the cursor session was not evaluated** (`atSession.evaluated === false`):
  the section already renders the honest no-eval line, and the server could only answer 404
  `no_evaluation_at_session`. Gating on a served record fact is a structural read, not a
  derivation. Likewise no fetch while the trace payload is loading/errored (the block mounts
  inside the evaluated branch).
- **D-e — `ComponentTrailEntry` gains `instanceId: string`.** THE integration gap this plan
  closes for M14.2b: the value-tap `component_path` is the chain of ComponentRef INSTANCE ids,
  but the trail today records only `(componentId, version)`. All three producer sites know the
  instance id and will supply it (Task 4).
- **D-f — Inner-node "At session" (M14.2b) is VALUES-ONLY.** Trace-event rendering for inner
  nodes (and the known one-level-flatten limitation) stays out of M14.2 scope; the inner
  section shows the cursor date, the no-eval line when apt, and the `ValueBlock`.
- **D-g — Value tests live in a NEW file** `web/src/components/Inspector.values.test.tsx`
  (deliberate refinement of the M14 plan's "extend Inspector.atsession.test.tsx"): a dedicated
  file keeps the existing at-session suite untouched, and that untouched suite passing is the
  additivity evidence. Mock the API-client module (keep `errorMessage` real via
  `importOriginal`, stub `getNodeValue` per test) — `useFetch` imports `errorMessage` from it.

## Task 1 — `getNodeValue` in the API client

**Files:** modify `web/src/api/client.ts`; test `web/src/api/client.test.ts` (extend, following
its existing per-endpoint patterns).

**Spec:** one new function, pure transport + typing (no domain types declared here — import
`NodeValueResponse` from `@quantize/quantize-api` and add it to the existing type-import list):

```ts
// The Node Value Tap (M14.2): the value a node's output port produced at a session — served,
// recompute-on-demand (`GET /v1/runs/{id}/values`). `componentPath` is the enclosing
// ComponentRef INSTANCE ids, outermost first, comma-joined in the query (segments are
// schema-constrained to ^[A-Za-z0-9_]+$, so the join is unambiguous); omitted for a top-level
// node. `outputPort` may be omitted only when the node has one output port — the server 422s an
// ambiguous omission, and the caller renders that served error rather than guessing.
export function getNodeValue(
  runId: string,
  address: {
    nodeId: string
    sessionDate: string
    componentPath?: readonly string[]
    outputPort?: string
  },
): Promise<NodeValueResponse> {
  const query = new URLSearchParams({
    node_id: address.nodeId,
    session_date: address.sessionDate,
  })
  if (address.componentPath !== undefined && address.componentPath.length > 0) {
    query.set('component_path', address.componentPath.join(','))
  }
  if (address.outputPort !== undefined) {
    query.set('output_port', address.outputPort)
  }
  return request<NodeValueResponse>(
    `/v1/runs/${encodeURIComponent(runId)}/values?${query.toString()}`,
  )
}
```

**Tests (RED → GREEN):**
1. Full address → exact URL asserted (`node_id`, `session_date`, `component_path=a,b`,
   `output_port` all present, run id percent-encoded).
2. Top-level, omitted port → URL carries NEITHER `component_path` NOR `output_port` (an empty
   `componentPath` array also omits it).
3. Non-2xx with a served `{code, message}` body → throws `ApiClientError` with that code/status
   (mirror the existing client error tests).

**Steps:** write tests → RED → implement → GREEN → typecheck → commit
`feat(m14.2a): getNodeValue client function`.

## Task 3 — M14.2a: the `ValueBlock` in the Inspector "At session" slot

### ValueBlock component (module-private in Inspector.tsx)

```tsx
function ValueBlock({
  runId, sessionDate, nodeId, componentPath, ports,
}: {
  runId: string
  sessionDate: string
  nodeId: string
  /** Enclosing ComponentRef instance ids, outermost first; empty at top level. */
  componentPath: readonly string[]
  /** The node's LISTED output-port names (catalog / exposed_outputs); empty = unknown → omit. */
  ports: readonly string[]
}): ReactElement
```

- **Port selection state:** `useState<string | undefined>(ports[0])` — `undefined` when `ports`
  is empty (D-c). Callers render `<ValueBlock key={...} />` keyed by the node address so the
  state resets per node (the ParamForm remount precedent). Render a
  `<select aria-label="output port">` ONLY when `ports.length > 1`; a single port renders as a
  static label; zero ports render no port UI (the response's own `output_port` labels the value).
- **Fetch:** `useFetch(() => getNodeValue(runId, { nodeId, sessionDate, componentPath, ...(port !== undefined ? { outputPort: port } : {}) }), [runId, sessionDate, nodeId, componentPath.join(','), port])`
  — note `componentPath.join(',')` in deps (array identity is not a stable key). Changing the
  selector fires a NEW request for that port only; no prefetch of unselected ports, no cache.
- **States:** loading → `<p className="inspector__empty-note">Loading value…</p>`; error →
  `<p className="inspector__at-error" role="alert">{error}</p>` (the served message verbatim —
  `engine_drift`, `dataset_mismatch`, `ambiguous_output_port` etc. all render whatever the
  server said, the honest-refusal pattern); data → the value rendering below.

### Value rendering (all fields verbatim, served order, `String(value)` for numbers/booleans)

Header line: the resolved port from the RESPONSE — `out {data.output_port}` (served, so the
defaulted-port case labels itself). Then switch on `data.value_summary.kind` (TS narrows the
union):

- `scalar`: one row — the `dtype` label + `String(value)`.
- `asset_set`: `{count} members` line + a `<ul>` of `members` in served order.
- `cross_section`: `{present_count} of {domain_count} assets` line; when `dtype === 'Number'`,
  min/max rows rendered only when non-null; when `dtype === 'Boolean'`, `true_count`/
  `false_count` rows only when non-null (never min/max labels); a `Missing: {missing.join(', ')}`
  line only when `missing` is non-empty; then the asset-values table.
- `time_series`: `{asset_count} assets · {total_points} points` line; a window row
  `{first_date} → {last_date}` only when `window` is non-null; then `series_preview` (when
  present): per asset a labeled group listing its `points` as `date value` rows — served order,
  ALL served rows rendered (the ≤64/asset cap is server-enforced; the client never truncates
  further), an empty-`points` asset renders its label with zero rows.
- `portfolio_targets`: the asset-values (weights) table, then `Weight sum: {String(weight_sum)}`
  and `Cash: {String(cash)}` rows — SERVED values only, never recomputed.

Asset-values table (shared for `cross_section`/`portfolio_targets`): a small `<table>` of
`asset` / `String(value)` rows in served order, rendered only when `asset_values` is present
and non-empty.

Provenance footer, driven by the served flag (a structural read):
`data.provenance.captured === false` → "Recomputed on demand from the run's pinned inputs" +
the `dataset_fingerprint` in a `<code>`; `true` → "Captured at run" (future-proof; the flag is
the contract's seam).
