# M9 — API Boundary + Strategy & Component Versioning: Implementation Contract

> Status: PLAN ONLY (authored during the pre-M9 remediation sprint; no API code exists yet).
> Authoritative scope: docs/MVP_PLAN.md §M9. This document pins the contract decisions the
> pre-M9 audit and plan reviews identified so M9 implementation starts with zero open
> architecture questions. Constraints inherited from CLAUDE.md invariants 5/6 (no business
> logic in the frontend; JSON DTOs only, no pandas/numpy/Python objects in contracts).

## 1. Endpoints (MVP_PLAN §M9 deliverables)

FastAPI, synchronous v0: validate IR · run backtest · run forward replay · fetch results ·
fetch traces · save strategy version · load strategy/version · create/list/load
`ComponentDefinition`s.

## 2. Validation endpoint — run-faithful, location-preserving

- The endpoint runs the FULL run-blocking preflight, not a subset: structural
  (`validate_strategy_document`), semantic (`validate_strategy_semantics`), component
  resolution (`resolve_strategy_components`), top-level component wiring (exposed-port
  existence/compatibility, exposed-input connectivity, and AMBIGUOUS FAN-IN — which is checked
  ONLY in that layer), and the terminal rule (exactly one `output.target_portfolio`).
- Composition lives in ONE extracted function that `evaluate_strategy`'s preflight also calls
  (no second implementation — plan-review finding; the seam already exists as the evaluator's
  preflight block, evaluate.py ~:425-464). Extraction mechanic (final-review finding 1.a): the
  shared function must return the NATIVE diagnostic shapes (loc-bearing structural/semantic +
  node-path RuntimeDiagnostics) and `evaluate_strategy` performs its own down-conversion to
  `RuntimeDiagnostic` at ITS call site — a naive lift-and-shift of the current inline block
  would reproduce the loc loss it exists to avoid.
- The response DTO carries BOTH diagnostic shapes, never flattened through the evaluator's
  `RuntimeDiagnostic` conversion (which drops `loc`):
  - loc-bearing: structural `StructuralError.loc` + semantic `SemanticDiagnostic.loc`
    (document-position tuples the editor can highlight);
  - node-path-bearing: resolution/wiring/terminal `RuntimeDiagnostic` (`node_path` +
    `subject`).
- Stable ordering: the existing deterministic sort per layer; layers reported in preflight
  order. Unsupported `schema_version` → 422 with the structural diagnostic; unparseable JSON →
  400; validation findings themselves are a 200 with `ok: false` + diagnostics (validation is
  the service, not the failure).

## 3. Database lifecycle — per-request, never process-global

- One `Database` (+ repositories) per request/operation: open → migrate (idempotent, cheap) →
  work → explicit close. No process-global SQLite connection; no cross-request handle reuse.
- Failure semantics are already structured at the boundary (pre-M9 D): `database_locked`
  (busy_timeout 5000 ms default), `integrity_violation` → domain `artifact_conflict`,
  `corrupt_database`, and every transaction failure path leaves the handle un-poisoned.
- HTTP mapping of stable codes: `artifact_not_found` → 404; `artifact_conflict` → 409;
  `database_locked` → 503 (retryable); `corrupt_artifact`/`corrupt_database` → 500 (operator);
  `invalid_artifact`/`unsupported_artifact_version` (client-submitted content) → 422;
  `unsupported_database_version` → 500 — the SERVER's own database file is newer than the
  code: an operator/deployment condition, never a client error (final-review finding 3.a).

## 4. Run identity — server-governed

- The server mints `run_id` (uuid4) — client-supplied run ids are never persisted authority
  (the engine raises on non-UUID input by contract; the endpoint must not forward client
  garbage into that raise).
- Forward replay: the run_id is minted ONCE per forward run and REUSED on resume
  (`ForwardCheckpoint.run_id`) — never re-minted per request.
- Idempotent submission stays content-addressed: duplicate `run_id` with byte-identical
  record+trace is a no-op; divergence is `artifact_conflict` (existing M7 semantics).
- Author-chosen content keys (`strategy.id` + version, `component_id` + SemVer) remain
  references with conflict-on-divergence — correct as-is.

## 5. Forward-replay endpoint semantics (v0)

- Bounded, run-to-exhaustion within one request: `last_session` REQUIRED (the M8 bounded-replay
  contract), returns the same run DTO as backtest, persists via `save_run` (which fails closed
  on non-exhausted peeks).
- Per-session interactive stepping with durable checkpoints is an explicitly deferred additive
  layer; `ForwardCheckpoint` is the preserved seam. Do NOT expose a stateful stepping handle
  across HTTP requests in M9.

## 6. Results / traces DTOs

- fetch-results returns the `PersistedRunRecord` envelope (format 2) as-is — it is already the
  plain-JSON contract shape, including `input_provenance`.
- fetch-traces returns the stored seq-ordered event stream (optionally date-filtered via the
  existing `load_trace(session_date=...)`); responses can be large — size note recorded;
  streaming/pagination is deferred with streaming trace persistence.

## 7. Input provenance exposure

- Runs expose `input_provenance` verbatim plus a server-computed convenience flag:
  `replay_verifiable = (status == "recorded")`.
- The API must NOT over-claim: "verified replayable" means "IF data matching the recorded
  dataset/calendar fingerprints is re-supplied, the deterministic engine reproduces the run" —
  the server stores hashes, not bytes, and cannot re-run on its own. Legacy runs (migrated
  format-1) report `status: "unknown"` — explainable, not replay-verifiable.
- Replay verification endpoint (optional in M9): given a run + candidate dataset, report
  `input_provenance_mismatches` (`dataset_mismatch` / `calendar_mismatch` /
  `unknown_provenance`) before attempting an engine comparison. Completeness note: full replay
  additionally needs the pinned strategy document (stored), the window (stored), and the
  INITIAL STATE (a `SessionEngine` constructor input, only implicitly recoverable from
  `valuations[0]`) — the verify endpoint must take initial state alongside re-supplied data.
- Upload flow: an uploaded dataset is content-named by `dataset_fingerprint` (sha256); the
  same bytes re-uploaded resolve to the same identity. A dataset REGISTRY/store is an M9
  implementation piece, not a schema question — the fingerprint functions exist.

## 8. Strategy identity & versioning (terminology pinned — pre-M9 G2/G4)

Four distinct identities; never collapse them into one hash:

| Identity | Meaning | Exists today |
|---|---|---|
| Content/artifact identity | `content_hash` of the EXACT persisted bytes | yes (M7) |
| Semantic identity | execution-relevant projection (`semantic_projection`; ui.* excluded) | function exists; NO stored hash |
| Strategy version | author-chosen `(strategy.id, version)` key | yes |
| Lineage | `forked_from` (`StrategyForkRef`) | yes |

- Save-as-new-version: client submits version N+1 (+ `forked_from`); server conflicts on
  `(id, version)` byte divergence. Dedupe is on EXACT BYTES, not semantics — a ui-only edit
  under the same version is a 409 (deliberate, documented product decision; the client bumps
  the version or replays the stored bytes). No stored semantic hash in M9 — no endpoint needs
  one; `semantic_projection` is computable on demand server-side if an "is this semantically
  identical?" affordance is ever wanted.
- Exposed-param `binds_to` decision (G4): schema-less registered nodes accept unverified
  exposed-param names until implementation-time validation (there is no authoritative
  param-name set without a schema); every v0 core node ships a strict `parameter_schema`, so
  the gap is unreachable today; `parameter_schema` becomes MANDATORY for future user-authored
  node types (a founder-reviewed restriction when that milestone arrives).

## 9. Concurrency & request isolation

- No mutable request-global state (verified: `build_core_catalog()` is per-call; registries are
  instance-scoped; `SessionEngine` holds all run state; the C1 memo and C2 cursor are
  runner-local by construction).
- Catalogs/services: immutable or request-scoped. Two concurrent runs in one process interact
  ONLY through the per-request `Database` handles (SQLite serializes writers; `database_locked`
  is the structured back-pressure).
- Synchronous v0 boundary: a run executes within the request. Async job execution is an
  additive later layer (job table + worker) — no seams needed now beyond the pure-function
  engine API that already exists.

## 10. Hosted evolution (preserved path, not work)

Browser → FastAPI → domain/runtime services → persistence protocols → SQLite development
adapter → later PostgreSQL adapter. NOT in M9: PostgreSQL, auth, users/workspaces, object
storage, background workers, async jobs, deployment infrastructure.

## Remaining M9 decisions (for implementation time, none blocking)

- FastAPI request-model layer: hand-written Pydantic DTOs vs re-using IR models directly
  (lean: reuse `StrategyDocument` for submission; wrap responses in thin envelopes).
  BINDING REQUIREMENT (final-review finding 7.a; MVP_PLAN §M9 risk + CLAUDE.md invariant 4):
  any NEW response/request envelope models are a cross-language contract and must flow through
  the codegen pipeline (schema + generated TS) with contract tests and the CI staleness gate —
  hand-maintained TS mirrors of API DTOs would reintroduce exactly the drift M9 exists to
  prevent.
- Dataset upload storage shape (bytes in SQLite TEXT vs file-backed) — small, isolated choice.
- Whether the validate endpoint also returns the resolved warm-up total (cheap, useful to the
  editor; additive).
