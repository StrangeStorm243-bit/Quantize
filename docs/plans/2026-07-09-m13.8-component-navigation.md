**Goal:** Replace the modal `ComponentDrawer` with in-canvas, read-only breadcrumb navigation
into component internals (arbitrary depth), restore marquee selection feeding the existing
extraction flow, and close the M13.7 hook so in-component trace rows navigate the breadcrumb to
the right nesting level.

**Architecture:** Every change is a projection change. The main `Canvas` gains a *component-view
mode*: when an App-owned trail (`ComponentTrailEntry[]`) is non-empty, it projects the trail
tip's `ComponentDefinition.implementation.graph` through the **same `toFlow`** and renders it
structurally read-only (no dispatch handlers passed). The drawer's `implementation.kind` gate
moves into this view. Trail state lives in the App (Inspector's "Enter component", canvas
double-click, and trace rows all push onto it). Marquee selection is re-enabled and mirrors into
the **existing App-owned extraction selection set** — the two-phase-commit extraction flow and
`ExtractDialog` are untouched.

**Tech stack:** unchanged — React 18 + `@xyflow/react` 12, Vitest + Testing Library. No new
dependencies.

**Branch:** `feat/m13.8-component-nav`, stacked on `feat/m13.7-debug-loop` (M13.7.5 commits
`b044a71`+`c91340f` — delivered to main via PR #20). Retarget/rebase onto main once PR #20 merges.

## Tasks

### Task 1: Shared trail type + trace-path resolver (pure, TDD)

Files: modify `web/src/document/flow.ts` (append), test `web/src/document/flow.test.ts` (append).

Add:

```ts
export interface ComponentTrailEntry { componentId: string; version: string }

export function resolveTrailFromPath(
  doc: Pick<StrategyDocument, 'nodes' | 'component_refs'>,
  componentPath: readonly string[],
  components: ReadonlyMap<string, ComponentDefinition> | undefined,
): ComponentTrailEntry[]
```

Walks doc → definition → definition over the served trace `component_path` (ComponentRef
INSTANCE node ids, outermost first). Returns the LONGEST resolvable prefix — an unknown node/ref
stops the walk; a definition cache miss (or non-`graph` kind) stops it AFTER the entry the ref
alone proves (the view ensures + loads that tip). Pure lookup; nothing fetched.

Tests: full 2-level resolution; cache-miss prefix (ref resolves level 1, walk stops);
plain-node/unknown-ref element → walk stops; empty path → `[]`.

### Task 3: Canvas component-view mode (core task, TDD)

Files: modify `web/src/components/Canvas.tsx`; create `web/src/components/Canvas.navigation.test.tsx`
(mock style of `ComponentDrawer.test.tsx`/`Canvas.extraction.test.tsx`); delete
`web/src/components/ComponentDrawer.tsx` + its test **in this task** (replacement coverage lands
same-slice); minimal `App.tsx` edit removing the drawer so the suite stays green.

New Canvas props:

```ts
componentTrail?: ComponentTrailEntry[]          // empty/absent = strategy editing view
onEnterComponent?: (entry: ComponentTrailEntry) => void
onNavigateToDepth?: (depth: number) => void     // breadcrumb clicks + Esc
componentSelectedNodeId?: string | null         // trace→breadcrumb emphasis inside component view
```

Behavior (one test per line):
1. Non-empty trail + tip def cached (kind `graph`) → projects
   `toFlow({ ...def.implementation.graph, component_refs: def.component_refs }, catalog, defs)`;
   breadcrumb `Strategy ▸ Name v1.0.0`.
2. Nested trail (2 entries) → deepest def rendered, two crumbs.
3. Structurally read-only: `nodesDraggable/nodesConnectable/elementsSelectable === false`,
   `deleteKeyCode === null`, and `onConnect/onNodesDelete/onEdgesDelete/onNodeDragStop`
   **undefined** (handlers passed conditionally — absent, not no-op).
4. Tip def not cached → "Loading component…" + `ensure` called (every trail level + the tip's
   nested refs — port the drawer's effect).
5. Tip kind ≠ `graph` → explanatory notice, no internal graph (the preserved gate).
6. Crumb click → `onNavigateToDepth(depth)`; Escape → `onNavigateToDepth(trail.length - 1)`
   (window keydown effect active only in component view).
7. Component view: StageStrip absent (Breadcrumb takes the strip slot); quick-add double-click
   and drop inert; `CanvasChromeContext.resolvable === false` (Data Source card placeholder rule).
8. Double-click a ComponentRef node → `onEnterComponent` with its pinned `(componentId, version)`
   (strategy view AND component view; resolve against the current view's refs; disabled in
   extraction mode). Via RF `onNodeDoubleClick`.
9. `componentSelectedNodeId` marks that projected node `selected: true` in component view.
10. Empty trail: existing Canvas tests stay green untouched.

## Risks / review focus

- Read-only must be structural (handlers absent, not disabled flags alone) — asserted on captured
  ReactFlow props.
- `onSelectionEnd` vs `onSelectionChange`: the re-seed effect programmatically sets
  `selected: true`; only `onSelectionEnd` + instance read avoids a selection-echo loop.
- The App→AppShell provider split touches every App test's mock surface — do it early in Task 4
  and run the whole suite before proceeding.
- Trail entries are pinned `(componentId, version)` against an immutable cache-forever store —
  the only degraded states are "not loaded yet" and "non-graph kind", both rendered honestly.
