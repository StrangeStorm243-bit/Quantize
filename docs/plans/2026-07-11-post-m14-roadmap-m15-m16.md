# Post-M14 Roadmap — M15/M16 Direction (2026-07-11)

> **Directional document, not an implementation plan.** M15 and M16 below are *likely* next
> phases with their rationale, decision inputs, and boundaries — deliberately without slices,
> file lists, or acceptance tests. Each phase, when its time comes, gets its own
> brainstorm → design → plan cycle and its own founder approval. Nothing here authorizes
> building anything.
>
> **Primary decision inputs, in order:** (1) §13 external validation findings (3–5
> quant-literate testers walking the journey — running first, per M14 design D-3);
> (2) M14 execution experience (including whether any capture flip-trigger fires);
> (3) the founder's read of which product axis is the binding constraint.

## 1. Where the product stands after M14 (assuming it lands as designed)

The spine is proven end-to-end: versioned JSON IR → one engine/evaluator → structured traces
→ a visual IDE in which structure (M13), decisions (M13), and dataflow values (M14) are all
legible. The 11 MVP success criteria in `docs/PRODUCT.md` are met or instrumented, except
that criterion 11 (a *documented path* to custom mathematical/Python components) remains
documentation-only — deliberately.

What the *vision* still lacks, by axis:

| Axis | State after M14 | Gap |
|---|---|---|
| **Vocabulary breadth** (statistics, modeling, optimization, eventually ML) | The v0 node set covers exactly the two reference strategies | A user whose strategy needs volatility, z-scores, or regression cannot express it |
| **Data reality** | Fixture + uploaded local datasets, daily bars | A user's own richer data (more history, more assets, other fields) is the first thing a real tester brings |
| **Authoring depth** | Components are immutable, extractable, inspectable — but not editable/forkable/re-versionable from the UI | "Modify a component" currently means editing the strategy graph, not evolving a shared definition |
| **Behavior over time** | Values tappable per session | No per-node series across a run (the first capture-at-run consumer) |
| **Execution reality** | Backtest + paper replay, honestly framed | Live/broker stays a labeled-deferred seam — intentionally out of this roadmap horizon entirely |

## 2. M15 (likely): Statistical vocabulary via a third reference strategy

**Thesis:** the vision promises "high-level strategies built from statistics, mathematics,
modeling, optimization" — but scope discipline (CLAUDE.md) forbids adding node types "to look
broad." The resolution is the pattern that built the entire v0 vocabulary: **pick one
concrete, founder-approved third reference strategy and add only the general-purpose nodes it
forces.**

- **Candidate strategy shapes** (choose ONE at design time, informed by §13 testers' own
  strategies): volatility-targeted momentum (forces a rolling-volatility node and a
  scale-by-risk weighting node); a z-score mean-reversion rotation (forces rolling mean/std
  and a z-score transform); a dual-momentum variant (may force nothing new — a useful
  null result that would redirect M15).
- **What M15 would prove:** the registry seam under growth — that adding statistics nodes is
  "one self-contained registration" with the full test battery (correctness, missing data,
  alignment, warm-up, look-ahead) and descriptor-driven UI metadata (role sentence, formula,
  category) arriving through the existing M13 machinery with **zero** frontend changes.
- **Boundaries it must keep:** no node added without the concrete strategy needing it; no
  stateful nodes unless the strategy genuinely forces one (and then the cadence/checkpoint
  rules in CLAUDE.md apply and the M14 capture flip-trigger 1 fires — a big, explicit
  decision, not a side effect); the strategy composed from primitives, never special-cased
  (the standing off-track signal).
- **When M15 would NOT be this:** if §13 says testers can't get *their data* in (→ M15
  becomes data ingestion), or that they don't trust the numbers (→ M15 becomes
  verification/transparency work), vocabulary waits.

## 3. M16 (likely): Component authoring — edit, fork, re-version

**Thesis:** components are the product's strongest architectural object (immutable,
versioned, compositional) but authoring them is one-way: extract, then inspect read-only.
"Modify a component and save the result as a new version" is MVP journey step 5 and the
seam M13.9 deliberately left clean (nothing in the current UI forks or edits a definition,
and invariant 8 keeps definitions immutable *per version*).

- **What M16 would add, conceptually:** open a component definition as an editable document
  (the same canvas machinery, per M13's document-centric shell) → save as a **new pinned
  version** (never mutating the old one) → an explicit strategy-side upgrade affordance for
  a `ComponentRef` (v1 → v2 as a reviewed, per-instance choice — never automatic). Forking
  (`forked_from` provenance already exists in the IR) is the same flow with a new identity.
- **What it would prove:** the reuse story end-to-end — build, share-with-yourself, evolve,
  upgrade consumers deliberately. This is also the last unproven MVP journey verb.
- **Boundaries it must keep:** invariant 8 untouched (immutability per version is the
  feature, not the obstacle); no marketplace/publishing/licensing surface (deferred paths);
  recursion rejection and pinning semantics unchanged.
- **Ordering note:** M16 after M15 is deliberate — a richer vocabulary makes component
  evolution *worth* exercising (e.g. Momentum Rank v2 with a volatility filter). If §13
  testers hit the authoring wall harder than the vocabulary wall, swap M15 and M16.

## 4. Standing candidates that stay parked (and what would un-park them)

- **Value-over-time + capture-at-run:** parked until an M14 flip-trigger fires (design §7).
  The first real demand will likely be "show me this signal across the run" — bring that to
  the founder as the capture milestone's opening evidence, not as an M14.x patch.
- **Custom math / sandboxed Python nodes (success criterion 11):** stays a documented path.
  Un-parked only by repeated §13-tester demand that the built-in vocabulary (post-M15)
  cannot satisfy — and then as its own security/ops-scoped design, behind the registry.
- **Optimizers / `Matrix` type:** un-parked only by a concrete strategy needing one
  (ARCHITECTURE.md §7 reserves the seam).
- **Real data providers, live/broker execution, multi-user, marketplace:** remain
  deferred-not-foreclosed seams. Nothing in M15/M16 may quietly build toward them.

## 5. Decision framework (apply when §13 results are in)

| §13 signal | Next-phase implication |
|---|---|
| "I understand the machine, but it can't express *my* strategy" | M15 as scoped above; pick the third reference strategy from the tester's own rules |
| "I can't get my data in / don't trust fixture-scale results" | Data-reality phase displaces M15; vocabulary waits |
| "I built it, but evolving/reusing components is the wall" | Swap M15 ↔ M16 |
| "Show me the signal over time, not per session" | Open the capture-at-run milestone (flip-trigger 2) before or alongside M15 |
| Legibility complaints persist (structure/values still unclear) | M14.3 (edge-hover dataflow) un-gates; possibly further legibility work before any new axis |
| Thesis-level contradiction (visual IDE isn't the tool they want) | Stop; founder-level product decision before any milestone |

## 6. What this roadmap refuses to do

No dates, no estimates, no slices, no acceptance tests for M15/M16 — those would be fake
precision ahead of §13 evidence. No scope beyond the two named phases plus the parked list.
The MVP's non-goals (`docs/PRODUCT.md`) and CLAUDE.md's scope discipline apply to every row
above; any phase that seems to require breaching them goes to the founder as a question, not
a plan.
