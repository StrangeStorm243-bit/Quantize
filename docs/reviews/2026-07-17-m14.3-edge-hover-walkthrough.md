## Post-review fix pass (2026-07-18, on PR #29)

The external review found one merge-blocking UI defect and one Low, both fixed:

1. **High — readout clipped in the default strategy view.** The extraction toolbar sits above the
   canvas inside the overflow-hidden center region while `.canvas` was `height: 100%` — so in the
   strategy view (only there; the component view has no toolbar sibling) the canvas overflowed by
   exactly the toolbar's height and the bottom-anchored readout clipped under the dock. The
   original shots `m143-05` (hint clipped) and `m143-08` (alert mostly hidden) show it — an
   honest miss in the first walkthrough pass. **Fix:** `.app-region--center` is now a flex column
   and `.canvas` fills the REMAINING height (`flex: 1 1 auto; min-height: 0`). **Browser-layout
   regression, re-run with the toolbar present** (real geometry, `getBoundingClientRect`,
   readout fully inside the center region):
   - success: readout bottom 715.4 ≤ region bottom 730.4 ✓ (`m143-11`)
   - pinned: the `Esc to release` hint fully visible ✓ (`m143-12`)
   - no-eval: honest line fully visible ✓
   - refusal (the worst pre-fix case): the full served-404 alert visible ✓ (`m143-13`)
   The canvas itself now bottoms out exactly at the region edge (730.40 = 730.40) with the
   toolbar present. No committed automated browser-layout harness exists in this repo (jsdom
   computes no layout — the same limitation the plan's Option-A amendment records), so this
   geometry check is the walkthrough-level regression; the CSS comments at both rules name the
   defect so a future edit can't silently revert it.
2. **Low — empty AssetSet digest rendered `0 members · ` with a dangling separator.** A valid
   empty selection now digests to just `0 members` — the members token is omitted when the set is
   empty (TDD: RED watched, `FlowReadout.test.tsx` asset_set suite).
