## Latency evidence (flip-trigger-3, threshold ~1s)

A **selective sample** of the `quantize.valuetap` INFO lines from the server console covering the
taps above (one line per tap quoted for readability; the full capture is 23 lines because every UI
tap double-fires under StrictMode — see Anomalies):

```
value tap run=e61d2c17-… addr=px   port=series  session=2025-07-31 elapsed_ms=68.0   (setup, curl)
value tap run=e61d2c17-… addr=sel  port=assets  session=2025-07-31 elapsed_ms=60.0   (UI, first fetch — semi-cold)
value tap run=e61d2c17-… addr=sel  port=assets  session=2025-07-31 elapsed_ms=52.8   (UI, second fire — StrictMode)
value tap run=e61d2c17-… addr=n43c03b4…/ret port=values session=2025-07-31 elapsed_ms=54.6
value tap run=e61d2c17-… addr=cap  port=targets session=2025-07-31 elapsed_ms=51.7
value tap run=932806df-… addr=n43c03b4… outcome=value_address_not_found session=2025-07-31 elapsed_ms=1.3  (refusal)
value tap run=e61d2c17-… addr=tp   outcome=value_address_not_found session=2025-07-31 elapsed_ms=38.6  (refusal)
```

**Verdict: PASS.** Across the full 23-line capture, every resolution attempt — successful taps and
honest refusals alike — completed well under the ~1s flip trigger. The slowest observed was
**130.6 ms** (a second-fire component re-tap); the first UI fetch of a given node is slightly slower
than its repeats (semi-cold — e.g. `sel` 60.0 ms first vs 52.8 ms on the double-fire), and warm taps
sat in a **~38–100 ms** band; address-not-found refusals were **1–2 ms**. Worst case is ~8× under
threshold; typical is 10–25× under. No tap approached 1 s. (These figures are from this
re-capture session; an earlier session showed the same profile with a slightly higher cold-start
first tap of 164 ms.)

## Anomalies / notes

- **StrictMode double-fetch (dev-mode artifact, not a product bug).** Every UI-driven tap emits
  **exactly two** `elapsed_ms` log lines; the single `curl` setup tap emitted **one**. This is React
  18 StrictMode double-invoking effects in the Vite dev server — an expected development artifact,
  not a production behavior. Recorded honestly rather than hidden. The value tap is idempotent, so
  the double fetch is harmless (identical result, no writes).
- The `cap` DTO's served `cash` field was additionally confirmed with a direct `curl`
  (`value_summary … "cash":0.0`); that `curl` accounts for one extra `elapsed_ms` line beyond the two
  UI double-fires for `cap` — not a third UI fire.
- **React Flow console warnings (pre-existing, benign).** On graph load, React Flow logs
  `Couldn't create edge for target handle id: "series"` / `source handle id: "values"` for the
  component instance's ports (error#008). These are unrelated to the value tap and did not affect any
  render; noted for completeness. The only other console error is a missing `favicon.ico` (404).
- **Supplementary corroboration (not committed).** Playwright MCP wrote accessibility/DOM snapshot
  `.yml` files and a console log into the untracked `.playwright-mcp/` directory during this session;
  they independently capture the same Inspector text (value blocks, refusal alerts, the no-eval note)
  quoted above. Per repo hygiene rule **D-25**, `.playwright-mcp/` is intentionally **not** committed
  — the reference is noted here only as extra evidence a reviewer can inspect locally.
- No empty-200 responses and no blank Inspector sections were observed at any step; every refusal
  rendered a specific, honest message.

## 4. Flip-trigger status (design §7)

The four capture-at-run triggers (`docs/plans/2026-07-11-m14-behavior-legibility-design.md:278–293`).
None fired; recompute-on-demand remains the M14 semantics and **capture-at-run stays inadmissible**
(any capture/persistence work in an M14 branch is a stop condition until one fires).

1. **Stateful nodes land — did not fire.** v0 ships no stateful nodes; every graph node is pure
   (design §5, approach B). The node registry is unchanged in M14 — no new node type was registered.
2. **Value-over-time UI becomes required — did not fire.** M14 surfaced no requirement for a
   per-node series column-scan across a run; the tap serves one session at a time. (Value-over-time
   remains a future decision, and it is the most likely first consumer of capture — see the
   deferral register.)
3. **Observed tap latency exceeds ~1 s on real datasets — did not fire.** Measured, not estimated:
   the attempt-level `elapsed_ms` instrument (`service.py:271`, emit at `service.py:290`) and the
   live 23-line console capture show worst 130.6 ms, warm 38–100 ms, refusals 1–2 ms — worst case

### 6b. Queued founder decision D-27 — canonical number display (awaiting founder decision — not implemented in M14.9)

Three number formatters coexist across the panels, so the same served number can render
differently depending on where you look:
- **Inspector** — `web/src/format.ts` `fmtValue` (`toFixed(4)` then trims trailing zeros and a
  dangling point; `format.ts:20–27`) → `126`.
- **TraceView** — `String(value)` raw (`web/src/components/TraceView.tsx:80`) → the full stored
  value.
- **ResultsView** — `fmt` = `toFixed(4)` untrimmed (`web/src/components/ResultsView.tsx:25–26`) →
  `126.0000`.

So the same number reads `126` in the Inspector vs `126.0000` in the Results panel, and a served
value below `0.00005` displays as a bare `0` in the Inspector (the verbatim value survives only in
the element's `title` attribute). This is a **presentation-consistency** decision, not a derivation
question — none of the three does arithmetic on the value (DoD box 9 holds).

*Options:* (A) unify on one shared display formatter in `format.ts`, consumed by the Inspector,
TraceView, and ResultsView, keeping the title-verbatim value everywhere; or (B) leave the three
per-surface formatters as they are (each arguably tuned to its panel).

*Recommendation:* **Option A** — one shared display formatter in `format.ts` consumed by all three,
with the full value always preserved in a `title`. It is presentation-only (no new derivation, no
contract change) and removes a cross-panel inconsistency a tester will notice. Deferred to the
founder; **not implemented in M14.9**.

## 7. Risks

- **Engine-drift window.** A tap recomputes against the engine as it is *now*. After an engine
  upgrade, tapping an old run can `409 engine-drift` until that run is re-run. This is a designed
  tripwire — the envelope-level trace cross-check (`14a00a1`) is what detects it; the remedy is to
  re-run the strategy, not to relax the check.
- **Recompute cost growth on real datasets.** The demo fixture is tiny; a full-history dataset will
  cost more per tap. Flip-trigger 3 is the standing instrument — the `elapsed_ms` log is already in
  place, so the ~1 s threshold would be *observed*, not guessed, if it were ever approached.
- **§13 external validation unconfirmed.** With no confirmed/recorded tester feedback, the M14
  legibility thesis remains **internally validated only** — the walkthrough proves the mechanism
  works, not that it answers a real user's "what is my strategy doing" question.
